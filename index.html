1. Project Overview

1.1 What We Are Building

Ambit Capital’s existing Research Portal (ambitresearch.co) will be retired and replaced with a brand-new, modern Research & Analytics Platform — built ground-up for institutional clients (mutual funds, AIFs, pension funds, family offices) who consume Ambit’s equity research on 270+ stocks across 47 sectors.

The new platform will deliver:

  • A clean, modern web portal (React, desktop-first, fully responsive)
  • A native-like mobile app for iOS and Android (React Native) — branded as A-ONE Research by IE Research
  • A robust editorial CMS to publish, edit, version and approve research reports
  • An integrated AI Summary & Audio capability — pre-generated at publish time, analyst-replaceable
  • A full HTML5 report reading experience — TOC, Focus Mode, Resume Reading, Trigger to Connect, Tags & Tag Cloud
  • Four brand-new modules: ESG / Corporate Governance analytics, Publishing Authors, Corporate Access, and Bespoke Work
  • SEBI-compliant infrastructure — immutable report archive, dynamic analyst disclosures, 5-year retention, full audit trail

1.2 Business Goals

  • Modernise client experience — the current portal’s UI is outdated (jQuery / server-rendered, no mobile); the revamp positions Ambit as a premier institutional research brand.
  • Deeper engagement & readership tracking — HTML reports enable read time, scroll depth, and interaction tracking (PDF cannot).
  • Drive analyst ↔ client connect — in-flow nudges (“Talk to the Analyst”) convert reading into conversations, lifting brokerage revenue.
  • Mobile-first reach — meet portfolio managers where they read; full iOS + Android app.
  • Operational efficiency — modern editorial workflow with checkout-locking removes bottlenecks in report production.
  • Compliance by design — SEBI Research Analyst Regulations 2014 compliant from day one.

2. Scope of Work

2.1 User Roles & Access

  • Admin — full access to all configuration, user management, content workflows, logs and analytics.
  • Editor / Production — access to the editorial CMS (review queue, WYSIWYG editor, publish, edit published).
  • Analyst (Author) — manages their own author profile; can replace AI-generated summaries; receives client Q&A and 1:1 requests.
  • External Client (End-User) — reads research, manages preferences, follows analysts, uses analytics, requests connects.
  • Sales — approves access requests, views readership analytics for their accounts.

Access is enforced through Role-Based Access Control (RBAC) configurable from the Admin module.

3. Functional Flow

3.1 End-User (Client) Journey

Step 1 — Land on portal: User visits the portal → if device has a valid cache token (issued in last 60 days), they are auto-logged-in → otherwise they enter their corporate email.
Step 2 — Authenticate: System emails an OTP → user enters OTP → new cache token is issued and stored on the device (max 3 devices per user).
Step 3 — Disclaimer & Cookies (first login only): Disclosure page shown (with selectable tickers + price charts) + Cookie preferences flyer.
Step 4 — Land on Dashboard: Recent Research feed (filtered by user preferences) + Key Reports + Upcoming Events + Search + Tags.
Step 5 — Discover: User searches a company / analyst / keyword or drills into the Coverage page (sector tree-map) or opens a research card.
Step 6 — Read Research: Full HTML report opens with TOC sidebar — user scrolls; AI summary slider available; audio playback option; Focus Mode toggle.
Step 7 — Engage: At half-scroll, “Talk to Analyst” trigger appears → user can submit a query or book a 1:1 slot via Calendly-like flow.
Step 8 — Save / Share: User bookmarks the report (resumable), copies short-link, follows the analyst.
Step 9 — Sales follow-up: Readership analytics surface engaged clients to the Sales / Account Manager → who reaches out to convert intent into a trade.

3.2 Editorial / Publishing Flow

Step 1: Publication System produces a research report (Word/PDF) and calls the portal’s Trigger API with document_id and ticker list.
Step 2: Portal pulls report content + financial data points via 3 fetch APIs → stores in PostgreSQL + S3.
Step 3: A worker queues the AI Summary + Audio generation → stored in document_summary table (content-hashed).
Step 4 — Editorial Review: Report appears in the editor’s “Awaiting Approval” queue → one editor checks it out (locked from others) → reviews and corrects in WYSIWYG.
Step 5 — Publish: On approval, report goes live → appears on Dashboards / Coverage / Research listings → followers get notified.
Step 6 — Post-publish edits: Admin can edit any published report (typo fixes, PDF replace) → AI summary auto-regenerates → audit log records every change.
Step 7 — SEBI archival: The published report — with disclosures injected — is written to the immutable S3 archive (Object Lock, WORM) for the 5-year retention window. Every read, edit and approval is appended to the audit ledger.

3.3 Admin Flow

Access Requests: New user requests access → appears in admin queue → admin approves / rejects → user receives onboarding email + first login OTP.
Configuration: Admin manages Sectors / Companies / Events / NSE↔Bloomberg mapping / Roles & Permissions.
Monitoring: Admin reviews Email logs, Integration logs, Readership analytics, Product usage tracking.

4.3 Word/PDF → HTML Strategy

The most challenging technical component. Recommended hybrid approach:

  1. Publication team exports report as HTML (Word built-in export) — ownership of structural correctness stays with the source.
  2. Our pipeline ingests this HTML, runs a sanitisation + normalisation pass (sanitize-html + custom DOM transformer), enforces our schema, injects TOC anchors, AI-summary block and audio anchors.
  3. For legacy reports we use Mammoth.js as fallback.
  4. Edge-case templates beyond the agreed canonical set are handled under T&M.

4.4 Security & SEBI Compliance Architecture

The platform is engineered to meet SEBI Research Analyst Regulations 2014 from day one. The compliance posture is built into the architecture, not retrofitted.

  • Immutable report archive — every published report is written to S3 with Object Lock (WORM) enabled; cannot be modified or deleted within the retention window.
  • 5-year minimum retention — lifecycle policy enforced at bucket level; tiered to S3 Glacier for cost.
  • Dynamic disclosure injection — per-ticker analyst disclosures are baked into the archived copy at publish-time; the disclosure version that was live at publish-time is preserved.
  • Append-only audit trail — every action (read, edit, approve, publish, login, token issue) writes to an append-only Postgres ledger plus CloudWatch Logs Insights; tamper-evident.
  • Encryption — TLS 1.3 in transit; AES-256 at rest (RDS, S3); KMS-managed keys.
  • Secrets management — AWS Secrets Manager for all API keys, DB credentials, third-party tokens.
  • Network security — AWS WAF (OWASP top-10 rules), rate limiting via Redis, private subnets for RDS / Redis / OpenSearch.
  • RBAC — least-privilege role model enforced at API and DB level; admin actions require step-up MFA.
  • SAST / DAST / SCA — in CI pipeline (Snyk / Semgrep / Dependabot); zero high-severity CVE on production deploy.
  • Penetration test — one external pen-test before Phase 1 go-live; remediation included.

5. Assumptions

  1. Publication System APIs — the existing Publication System will provide the Trigger API endpoint, plus 3 fetch APIs with stable JSON schemas, ≥99.5% uptime, retry semantics and idempotency keys defined.
  2. Word/PDF source — Ambit’s production team will provide a maximum of 8 sanctioned Word templates at kickoff. Any free-form analyst formatting / equations / embedded Excel objects beyond these templates may require T&M.
  3. AI provider & tokens — AI summary & audio TTS use a hosted LLM API (OpenAI / Azure OpenAI / Anthropic). Ambit holds the API key and bears monthly token / TTS consumption costs. We integrate, debounce, and content-hash to minimise spend.
  4. Cloud account — Production AWS account is owned and billed directly by Ambit; we deploy into it.
  5. Domains & certificates — Ambit provides domain, SSL certificates, email-sender domain (DNS access for SES configuration).
  6. Design assets — Final Figma designs (current revision) are the source of truth. Visual changes to designs post-kickoff follow Change Request process.
  7. Historical content — Pre-cutoff PDFs are served as-is (no HTML re-render). HTML rendering applies to reports published after go-live (or a mutually agreed cutoff).
  8. Compliance officer — Ambit assigns a SEBI Research Analyst compliance SPOC who signs off the disclosure engine, audit-trail design and retention policy.
  9. Editorial / analyst SMEs — available 4 hrs / week for clarifications on coverage taxonomy, HWAK scoring, editorial workflows.
  10. UAT — Ambit nominates UAT testers and signs off each milestone within 5 working days; otherwise milestone is deemed accepted.
  11. Calendly procurement — If 1:1 booking integration uses Calendly (or equivalent SaaS), Ambit procures and pays for the licence; we handle integration only.
  12. App Store / Play Store accounts — Ambit owns Apple Developer + Google Play Console accounts; we publish under those accounts. App-store fees borne by Ambit.
  13. HWAK / ESG data — Ambit’s research team provides the scoring formulae, weights, data sources and refresh cadence in writing during Phase 1A; we build to spec.
  14. Bloomberg / NSE data licences — Ambit holds all market-data licences. We integrate; data licensing fees are direct-to-Ambit.
  15. Browser / OS support matrix — latest 2 versions of Chrome, Edge, Safari, Firefox; iOS 15+, Android 10+. Older versions are best-effort.
